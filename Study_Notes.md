notebook. The expected solution must show that a model can be built, tracked,

The selected business problem is heart disease prediction using the UCI Cleveland
Heart Disease dataset. Each row contains patient clinical measurements such as
age, chest pain type, blood pressure, cholesterol, maximum heart rate, exercise
angina, number of colored vessels, thalassemia result, and a target label. The
original target has values 0, 1, 2, 3, and 4. For this assignment, it is converted
into a binary classification target:

- `0` means no heart disease.
- `1` means heart disease present, combining original labels 1 to 4.

The final system predicts whether a patient is likely to have heart disease and
returns both the predicted class and the probability of heart disease.

## 3. Data and EDA

The raw UCI file has no header row, so `src/preprocess.py` assigns the correct
14 column names. Missing values are represented by `?`, mainly in `ca` and
`thal`. The code converts these placeholders to `NaN` so sklearn imputers can
handle them correctly.

The required EDA evidence is generated by:

```bash
python -m src.eda
```

This creates:

- `plots/eda_class_balance.png`
- `plots/eda_missing_values.png`
- `plots/eda_numerical_histograms.png`
- `plots/eda_correlation_heatmap.png`
- `plots/eda_age_thalach_relationship.png`

These plots satisfy the FAQ requirements for histograms, correlation heatmap,
class distribution, missing value analysis, and feature relationship analysis.

## 4. Preprocessing and Feature Engineering

The project uses a single sklearn `ColumnTransformer` inside a full sklearn
`Pipeline`. This is important for MLOps because the exact same preprocessing is
used during training and inference.

Numerical features:

- `age`
- `trestbps`
- `chol`
- `thalach`
- `oldpeak`
- `ca`

Numerical processing:

- Missing values are imputed using median.
- Features are scaled using `StandardScaler`.

Categorical features:

- `sex`
- `cp`
- `fbs`
- `restecg`
- `exang`
- `slope`
- `thal`

Categorical processing:

- Missing values are imputed using the most frequent value.
- Categories are one-hot encoded.
- Unknown future categories are ignored to avoid inference-time failures.

This design prevents training-serving skew because the API receives raw JSON
features and the saved pipeline transforms them the same way training did.

## 5. Model Training and Selection

The assignment requires at least two classification models. This solution uses:

1. Logistic Regression as a strong, interpretable baseline.
2. Random Forest as a non-linear tree ensemble.

Both models use `GridSearchCV` with 5-fold cross-validation and ROC-AUC scoring.
The evaluation metrics are:

- Accuracy
- Precision
- Recall
- F1-score
- ROC-AUC

Run training with:

```bash
python -m src.train --model both
```

This command trains both required models, logs parameters, metrics, plots, and
model artifacts to MLflow, compares the runs using ROC-AUC, and saves the best
pipeline locally to:

```text
models/best_model.joblib
```

MLflow may print a security advisory about pickle or cloudpickle serialization.
That advisory is expected for sklearn model artifacts and does not mean the run
failed. Only load model files created by this project or another trusted source.

The model is served using FastAPI. The API loads the saved pipeline from
`models/best_model.joblib`.

Start the API:

```bash

```

Open Swagger UI:

```text
http://127.0.0.1:8000/docs
```

Test health:

```bash
curl http://127.0.0.1:8000/health
```

Test prediction:

```bash
curl -X POST http://127.0.0.1:8000/predict \
  -H "Content-Type: application/json" \
  --data @examples/patient_request.json
```

Test metrics:

```bash

```

The `/predict` endpoint must return:

- `prediction`: 0 or 1.
- `confidence`: probability of heart disease.
- `model_version`: current API model version.

## 13. Common Viva or Explanation Questions

Why use a pipeline?

The pipeline guarantees that preprocessing and prediction happen together. This
prevents a common production problem where training data is transformed one way
but live API data is transformed differently.

Why use MLflow?

MLflow records parameters, metrics, artifacts, and model files, making the model
selection process reproducible and auditable.

Why use Docker?

Docker packages the application, dependencies, model, and runtime into one image,
so the API runs consistently on different machines.

Why use Kubernetes?

Kubernetes demonstrates production deployment concepts such as replicas,
readiness probes, liveness probes, service exposure, and rolling updates.

Why use `/metrics`?

The `/metrics` endpoint allows Prometheus to scrape API performance and request
metrics, which supports monitoring in production.

Why use cross-validation?

Cross-validation estimates how well the model generalizes by training and
validating on multiple data splits instead of trusting one split only.

Why use ROC-AUC?

ROC-AUC evaluates ranking quality across classification thresholds, which is
useful for medical risk prediction where the threshold may be adjusted.
